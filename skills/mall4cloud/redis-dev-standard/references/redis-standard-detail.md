# Redis 完整规则整理

本文依据内部草稿规范 `REDIS-STD-V2026070301` 整理，不复制组织、人员、真实连接信息或可直接运行的生产删除代码。标签含义：

- **强制**：规范明确要求，变更默认必须满足。
- **推荐**：优先采用；偏离时说明原因。
- **需确认**：涉及版本、架构、容量或来源冲突，需 DBA/架构书面确认。
- **来源存疑**：原文存在冲突、过时示例或环境依赖，不可机械复制。

## 存储选型

- **推荐**：高 QPS、低延迟、容量可控且数据可重建时优先评估 Redis。
- **推荐**：低频大量数据、强事务和强持久需求优先使用 MySQL 等事实存储。
- **强制**：Redis 默认只作缓存，不是关键可靠消息队列，也不是强一致数据的唯一来源。
- **强制**：冷热数据分离，不把全部业务数据写入 Redis。
- **强制**：不同业务和不同热度数据按实例、集群或明确边界隔离。
- **推荐**：单实例 Key 数控制在 100 万以内；永久 Key 场景仍需容量评估，不能视为无上限。

## 数据结构

| 类型 | 适用场景 | 来源容量说明 | 新设计检查 |
| --- | --- | --- | --- |
| String | 简单 KV、计数、短文本 | 一处建议小于 1 MB | 按 BigKey 规则控制在 10 KB |
| Hash | 实体对象、关联属性 | field 少于 10000、整体小于 1 GB | 按 BigKey 规则不超过 5000 元素 |
| Set | 去重、标签、无序成员 | 元素少于 10000、整体小于 1 GB | 按 BigKey 规则不超过 5000 元素 |
| Sorted Set | 排行榜、权重列表 | 元素少于 10000、整体小于 1 GB | 按 BigKey 规则不超过 5000 元素 |
| List | 有序列表、非关键临时缓冲 | 长度少于 10000、整体小于 1 GB | 按 BigKey 规则不超过 5000 元素 |

- **推荐**：String 存数字最节省；长文本先评估压缩成本和其他存储。
- **推荐**：对象属性优先集中在 Hash，不拆成大量独立 Key。
- **强制**：List 生产与消费速率必须平衡，不作为需要可靠投递的消息队列。
- **需确认**：HyperLogLog、Geo、Lua/Scripting 等特殊类型在使用前与 DBA 沟通。

## Key 设计

- **推荐**：使用 `业务:模块:唯一标识` 的冒号命名空间，避免不同业务冲突。
- **推荐**：在保持语义清晰的前提下缩短高基数 Key，减少内存占用。
- **强制**：禁止空格、换行、单双引号和各类转义字符。
- **强制**：Key 不包含密码、Token、身份证件、手机号等敏感明文。
- **强制**：多租户和多环境数据必须有不可绕过的隔离维度。
- **推荐**：Cluster 中需要多 Key 原子或 Lua 操作时，按项目规范设计 hashtag，并验证 slot 分布。

## Value 与 BigKey

- **强制**：String 序列化后不超过 10 KB。
- **强制**：Hash/List/Set/ZSet 不超过 5000 个元素。
- **强制**：调用 O(N) 命令前同时限制成员数、返回数和字节量。
- **推荐**：高基数结构拆成多个有明确路由规则的小结构。
- **强制**：非 String BigKey 不直接同步 DEL；Hash/Set/ZSet 使用对应 SCAN 命令渐进清理，List 使用分段 LTRIM，或采用版本支持且经确认的异步删除。
- **强制**：不能依靠给 BigKey 设置短 TTL 来规避删除风险；自动过期仍可能造成主线程延迟。
- **推荐**：监控内存占用、序列化大小、元素数、延迟和热点访问。

## 生命周期

- **强制**：缓存 Key 设置 TTL，默认最长 14 天；超过 14 天或永久存储需 DBA 确认。
- **强制**：批量创建的 Key 打散过期时间，防止同一时刻集中失效。
- **推荐**：永久 Key 持续监控 idle time，并建立清理责任人和复核周期。
- **需确认**：业务依赖 Redis 原生过期语义时，读写分离拓扑可能产生行为差异；由 DBA 基于实际版本确认。
- **强制**：核心业务和连接操作捕获异常，失败路径不能留下不确定状态。

## 命令复杂度

- **强制**：MGET、MSET、HMGET 等批量命令单次不超过 100 个元素，总数据量不超过 10 KB。
- **强制**：HGETALL、HKEYS、HVALS 仅在 field 不超过 100 且整体不超过 10 KB 时使用。
- **强制**：大 Hash 使用 HSCAN 分段遍历，必要时拆分。
- **强制**：LRANGE、SMEMBERS、ZRANGE、SINTER 等遍历命令必须有小而明确的 N。
- **强制**：线上不直接执行 KEYS、FLUSHALL、FLUSHDB；使用有界 SCAN 或运维工具。
- **推荐**：原生批量命令用于同类型操作，Pipeline 用于减少多类命令的网络往返。
- **强制**：不能把 Pipeline 当作原子事务；失败时逐项处理结果。
- **推荐**：少用 Redis 事务，因为不支持传统数据库式回滚。
- **强制**：Cluster 事务内的 Key 必须位于同一 slot。
- **强制**：Lua 操作的全部 Key 通过 KEYS 传入，不在脚本中硬编码业务 Key。
- **强制**：Lua 全部 Key 位于同一 slot，脚本执行时间和返回体有上限。
- **强制**：MONITOR 仅由运维短时使用。
- **推荐**：不使用 SELECT 作为业务隔离方案。

## 客户端

- **强制**：无关业务使用独立实例或明确隔离的服务，不共享故障域。
- **强制**：使用项目当前管理的连接池客户端，不在业务代码中直接创建裸连接。
- **强制**：配置连接、命令和池等待超时，资源在所有路径正确归还。
- **强制**：异常日志不泄露凭据、完整 Value 或用户隐私。
- **推荐**：高 QPS 关键链路配置熔断、限流、降级和数据库回源保护。
- **强制**：实例使用强认证，公网或云环境按平台能力开启加密传输和网络访问控制。
- **强制**：凭据来自安全配置系统，不硬编码到代码、文档或命令行。

## 内存淘汰

策略选择必须结合全部 Key 是否有 TTL、数据可否重建、业务降级和托管平台能力：

- `allkeys-lru`：从所有 Key 中按近似 LRU 淘汰。
- `allkeys-random`：从所有 Key 中随机淘汰。
- `volatile-random`：只从带 TTL 的 Key 中随机淘汰。
- `volatile-ttl`：优先淘汰更早过期的 Key。
- `noeviction`：内存满后拒绝新增写入。

**来源存疑**：不得假设 `volatile-lru` 是所有环境默认策略；读取目标实例真实配置并由 DBA 确认。

## BigKey 定位与清理

- **强制**：生产 BigKey 扫描、热点分析、迁移和删除由 DBA/运维执行。
- **推荐**：使用平台批准的扫描与迁移工具，不在 Skill 固化特定第三方工具。
- **强制**：清理脚本可暂停、可限速、可观测，并处理并发写入和失败重试。
- **推荐**：Redis 4.0 及以上且版本/拓扑允许时评估 UNLINK。

按类型的渐进策略：

| 类型 | 渐进策略 |
| --- | --- |
| Hash | HSCAN 小批扫描，HDEL 小批删除，最后处理空 Key |
| List | LTRIM 分段截断，限制单次处理量，最后处理空 Key |
| Set | SSCAN 小批扫描，SREM 小批删除，最后处理空 Key |
| Sorted Set | ZSCAN 小批扫描，ZREM 小批删除，最后处理空 Key |

来源代码示例不应直接使用，原因包括裸连接、明文连接参数、逐元素同步删除、缺少限速以及无法安全处理并发写入。

## 缓存稳定性补充检查

来源未完整覆盖但实现必须明确：

- 缓存穿透：参数校验、短 TTL 负缓存或项目批准的过滤机制。
- 缓存击穿：互斥重建、单飞或逻辑过期。
- 缓存雪崩：TTL 抖动、分批预热、限流和降级。
- 热点 Key：监控、局部副本、数据拆分或请求合并。
- 一致性：数据库提交、缓存失效/更新顺序及失败补偿。
- 重试放大：限制重试次数，使用退避并保护事实数据库。

## 来源冲突处理

- 原文是草稿，不能替代已发布制度或 DBA 书面结论。
- 容量阈值冲突时，对新设计使用 10 KB、5000 元素和 100 元素批次作为保守检查。
- 关键可靠队列禁止使用 Redis；非关键临时 List 场景需单独评审。
