# MySQL 完整规则整理

本文依据内部草稿规范 `MYSQL-STD-V2026070301` 整理，不复制文档中的组织、人员或基础设施信息。标签含义：

- **强制**：规范明确要求，变更默认必须满足。
- **推荐**：优先采用；偏离时说明原因。
- **需确认**：涉及版本、架构、容量或来源冲突，需 DBA/架构书面确认。
- **来源存疑**：原文存在冲突或示例不合规，不可机械复制。

## 命名

- **强制**：库、表、字段、索引使用小写字母和下划线，名称与业务含义对应。
- **强制**：内部名称长度不超过 32 字符；MySQL 自身 64 字符上限不能替代内部限制。
- **强制**：禁止 MySQL 保留字，禁止以 `test` 为前缀。
- **强制**：临时库和临时表使用 `tmp_` 前缀，并附业务日期。
- **推荐**：废弃对象与归档对象按 DBA 管理的专用区域存放，确认无业务使用后再清理或迁移。
- **推荐**：不使用存储过程、触发器、视图、UDF、EVENT 等数据库内置业务逻辑。

## 库表基线

- **强制**：默认存储引擎为 InnoDB。
- **强制**：库、表、字段和客户端字符集统一为 `utf8mb4`；collation 跟随目标数据库基线。
- **强制**：除主键字段外，表和字段都添加 comment。
- **强制**：数据库不保存图片、文件等大二进制内容，只保存对象存储引用。
- **强制**：单库不超过 500 张表，单表不超过 50 个字段。
- **推荐**：字符字段为主的单表控制在 3000 万行内，整型为主的单表控制在 5000 万行内。
- **强制**：测试环境不得直连生产环境调试；生产环境不执行未经审批的压力测试。
- **推荐**：按冷热程度分级存储，历史冷数据进入归档流程。

## 表设计

- **强制**：前端业务不使用分区表；确有需要时按例外走架构评审。
- **推荐**：高频关键字段与大字段、低频字段进行冷热拆分。
- **需确认**：分表优先评估 HASH 或 RANGE；数字分片后缀从 0 开始。
- **需确认**：归档或时间分表后缀使用 `YYYY[MM][DD][HH]` 形式。
- **强制**：新业务表包含以下基础字段，并与项目公共模型保持类型一致：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `create_time` | 创建时间 |
| `create_user` | 创建人 |
| `update_time` | 修改时间 |
| `update_user` | 修改人 |

## 字段设计

- **推荐**：字段精简并选择满足真实范围的最小适配类型；允许为减少昂贵 JOIN 做经过评审的适度冗余。
- **推荐**：非负数值使用 UNSIGNED。
- **强制**：IPv4 如以数值形式保存，使用 `INT UNSIGNED`；IPv6 或兼容双栈时重新设计。
- **强制**：金额及其他精确数值使用 `DECIMAL`，不使用 `FLOAT`、`DOUBLE`。
- **需确认**：原文同时要求“整型统一 INT”和“按范围选最小类型”，新设计按真实范围与项目类型兼容性评审。
- **强制**：ENUM 只允许在末尾新增值，不改变既有值含义和默认值。
- **推荐**：优先用合理长度的 VARCHAR；TEXT/BLOB 必须拆到主键关联附表并走评审。
- **推荐**：大小写敏感或二进制内容可评估 VARBINARY。
- **强制**：VARCHAR 长度按实际字符上限设计，不使用无依据的大长度。
- **推荐**：仅年份使用 YEAR，仅日期使用 DATE，精确到秒的时间结合版本和时区要求选择 TIMESTAMP 或经确认的项目类型。
- **强制**：默认使用 NOT NULL 和明确默认值。
- **需确认**：NULL 是独立业务状态时必须保留真实语义，不能用 `0` 或空字符串造成歧义。

## 索引

- **强制**：每张表有主键；不使用数据库外键。
- **推荐**：默认使用无符号自增非业务 ID；项目已有分布式 ID 时保持一致。
- **强制**：单表索引总数不超过 5 个，联合索引字段不超过 5 个；例外需提交关键 SQL 和写放大评估。
- **强制**：普通索引使用 `idx_`，唯一索引使用 `uk_`，联合索引名称包含参与字段或清晰缩写。
- **强制**：不创建重复、冗余或无法服务查询的索引。
- **强制**：JOIN 字段类型完全一致并有索引。
- **推荐**：选择区分度高的列，联合索引中优先放置能有效缩小结果集且符合查询前缀的字段。
- **强制**：不为极低区分度字段单独建立普通索引。
- **需确认**：长字符串使用前缀索引时，原文限制前缀不超过 8 字符；必须用真实数据验证区分度。
- **推荐**：超长字符串检索可增加 CRC32/MD5 等哈希伪列，但必须处理碰撞并回查原值。
- **推荐**：利用联合索引最左前缀和覆盖索引，减少随机 IO 和回表。
- **推荐**：新增替代索引并验证稳定后，清理确定无用的旧索引。
- **推荐**：更新频繁表严控索引数量，大批量更新拆分执行。

## SQL 安全

- **强制**：使用 prepared statement 或框架安全参数绑定。
- **强制**：数值参数保持数值类型，字符串保持字符串类型，避免隐式转换。
- **强制**：动态表名、列名和排序字段使用服务端白名单，不接受原始用户输入。
- **强制**：INSERT 显式指定列；禁止 `INSERT INTO ... SELECT *`。
- **强制**：单条 DML 不更新多张表。
- **强制**：UPDATE/DELETE WHERE 条件必须有界且可命中索引；执行前验证影响行数。
- **强制**：SQL 生成逻辑拒绝空 WHERE、恒真条件和意外全表操作。
- **强制**：变更脚本不包含清空全表的 DROP、TRUNCATE 或等价操作。
- **强制**：DML 中不使用会话变量代替真实字段名。

## SQL 性能

- **推荐**：使用 IN 替代大量 OR，IN 参数不超过 500 个。
- **强制**：禁止 3 张及以上表 JOIN；减少 JOIN 和子查询，复杂逻辑拆分为有边界的小 SQL。
- **强制**：索引列不做数学运算或函数封装。
- **推荐**：批量 INSERT 减少交互，但根据行大小和锁影响控制批次。
- **强制**：批量读取分页处理，单次少于 500 行且结果集控制在 1 MB 内。
- **推荐**：可接受重复结果时使用 UNION ALL。
- **强制**：多表查询明确列名，不使用 `SELECT *`。
- **推荐**：避免在关键 SQL 中使用 RAND、SYSDATE、CURRENT_USER 等动态不确定函数。
- **强制**：禁止无保护的深分页；采用基于游标、主键或业务锚点的项目既有方案。
- **需确认**：高频前端业务不直接实时执行昂贵 `COUNT(*)`，可使用缓存、统计表或专用统计链路。
- **强制**：避免全表扫描；UPDATE/DELETE 条件、JOIN 关联、ORDER BY、GROUP BY、DISTINCT 涉及的访问路径必须整体评估索引。
- **推荐**：不需要排序的分组查询可结合版本验证取消额外排序。
- **强制**：避免前导通配符，例如 `LIKE '%abc'`。
- **强制**：新 SQL 避免 `NOT IN`、`!=`、`NOT LIKE`、`<>` 等负向条件；确需使用时必须验证计划和数据量。
- **强制**：新设计避免通过 `IS NULL` 检索；保留 NULL 语义的例外必须有适配索引和计划验证。
- **强制**：使用 EXPLAIN 检查访问类型、key、rows 和 Extra。
- **需确认**：出现 `Using filesort` 或 `Using temporary` 不自动等于错误，但必须说明成本、数据量和替代方案。

## 事务与运行环境

- **强制**：SQL 异常不能被吞掉，事务必须按项目机制回滚。
- **强制**：应用不得擅自修改生产数据库事务隔离级别。
- **推荐**：拆分大事务，降低锁持有时间、主从延迟和失败恢复成本。
- **强制**：生产业务从库不执行未经审批的临时统计分析；统计需求使用专用链路。

## 数据库变更

变更申请至少包含：

1. 申请人与所属项目。
2. 数据库类型、受影响环境、逻辑数据库或 schema。
3. 经批准的目标连接标识，不在 Skill、代码或聊天中记录真实密码。
4. 完整 SQL 脚本。
5. 计划执行时间和预计耗时。
6. 业务影响、锁与主从延迟风险。
7. 回滚脚本或无法回滚时的补偿方案。
8. 验证 SQL、预期影响行数和执行后观察项。

脚本要求：

- **强制**：DDL/DML 不携带库名前缀，由 DBA 切换目标库。
- **强制**：新表自增主键初始值为 1。
- **强制**：不使用反引号等不必要特殊符号。
- **强制**：新增字段默认不使用 `AFTER`，保持对数据库代理的兼容。
