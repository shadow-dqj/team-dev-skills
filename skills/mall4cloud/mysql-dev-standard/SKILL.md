# MySQL Dev Standard

## 适用范围

用于工作区内所有 MySQL 设计与变更，尤其是 Mall4Cloud 后端的表结构、MyBatis Mapper/XML、查询、索引、数据修复脚本和数据库工单。

规范源 `MYSQL-STD-V2026070301` 当前标记为草稿。实现时同时遵循目标仓库相邻代码和 DBA 已确认要求；若本 Skill、数据库实际版本、项目契约或 DBA 结论冲突，以更高优先级的安全约束和书面确认结果为准。

## 工作顺序

1. 确认目标独立仓库、Maven 模块、数据库、表、数据规模和读写路径。
2. 追踪 model、DTO/VO、Mapper/XML、Service、API 和调用方，避免只改 SQL。
3. 分类检查命名、字段、主键、索引、查询计划、锁、事务、兼容和回滚影响。
4. 对高风险 DDL/DML、例外设计、分片、归档和容量问题取得 DBA/架构确认。
5. 执行最小相关测试和 SQL 静态检查，并记录执行计划、影响行数和回滚方案。

## 必须遵循

- 库名、表名、字段名和索引名使用小写字母及下划线，名称见名知意；内部规范长度不超过 32 字符。
- 禁止使用 MySQL 保留字和 `test` 前缀；临时对象使用 `tmp_` 前缀并带业务日期。
- 默认使用 InnoDB 和 `utf8mb4`；除主键字段外，表和字段必须有业务含义明确的 comment。
- 每张表必须有主键。默认使用无符号自增非业务 ID 或项目现有分布式全局 ID，不使用数据库外键约束。
- 新业务表包含 `id`、`create_time`、`create_user`、`update_time`、`update_user` 五个基础字段；与现有公共模型类型保持一致。
- 金额、支付等精确数值使用 `DECIMAL`，禁止用 `FLOAT` 或 `DOUBLE`；图片和文件放对象存储，数据库只保存访问标识或地址。
- 字段默认定义 `NOT NULL` 并设置有业务意义的默认值。若空值具有独立业务语义，不得用 `0` 或空字符串混淆，需在评审中明确例外。
- JOIN 字段类型必须完全一致并有可用索引，避免隐式类型转换；不创建重复或冗余索引。
- 普通索引使用 `idx_<字段>`，唯一索引使用 `uk_<字段>`；联合索引名称反映参与字段。
- SQL 必须参数化或使用 prepared statement。不得把用户输入拼接为值、表名、列名、排序字段或条件片段。
- `INSERT` 显式列出字段；禁止使用全字段复制语句。单条 DML 不同时更新多张表。
- `UPDATE`、`DELETE` 必须使用有界且可命中索引的 WHERE 条件，执行前核对影响行数；不得依赖 `LIMIT` 掩盖非索引更新风险。
- 数据库变更脚本不携带库名前缀，不使用反引号，不包含清空全表的 `DROP`、`TRUNCATE` 或恒真条件。
- 变更工单包含目标范围、完整脚本、计划时间、业务影响、执行耗时预估和回滚或补偿方案。

## 设计与性能约束

- 单库表数量控制在 500 以内，单表字段控制在 50 以内；接近阈值时先做容量和拆分评审。
- 字符类型为主的单表控制在 3000 万行以内，整型为主的单表控制在 5000 万行以内；达到阈值前规划归档或分片。
- 单表索引总数控制在 5 个以内，单个联合索引字段不超过 5 个；优先高区分度、复合和覆盖索引。
- `IN` 参数不超过 500 个；批量读取分页处理，单次结果少于 500 条且结果集控制在 1 MB 内。
- 禁止 3 张及以上表 JOIN；减少子查询和复杂大 SQL，优先拆成有明确边界的简单操作。
- 索引列不做函数封装或数学运算；避免前导通配符、负向条件和 `IS NULL` 条件导致索引失效。
- 可接受重复数据时使用 `UNION ALL`；查询只选择业务需要的列，不使用多表 `SELECT *`。
- 对新增或变更 SQL 使用 `EXPLAIN`。出现全表扫描、`Using filesort` 或 `Using temporary` 时必须说明原因并取得评审确认。
- 深分页、实时总量统计和大批量写入必须采用项目已验证的替代方案，不直接把高成本操作压到主业务库。

## 跟随项目实现

- Spring 事务沿用项目 `@Transactional(rollbackFor = Exception.class)` 等既有机制；异常必须触发回滚，但不要在声明式事务中机械复制手动 `rollback`。
- MyBatis `@Param`、XML 占位符和 foreach collection 必须一致，动态 SQL 只使用安全参数绑定。
- 字段或索引变更同步检查 model、DTO/VO、Mapper resultMap、序列化契约、接口调用方和测试。
- 新增字段默认不使用 `AFTER`；确需调整物理列顺序时先确认当前代理和数据库环境是否允许。

## 需 DBA 或架构确认

- 分库分表、HASH/RANGE 分片、时间归档表、冷热分离和历史数据迁移。
- 超过容量、字段、索引、批量或结果集阈值的场景。
- TEXT/BLOB、超长 VARCHAR、前缀索引、哈希伪列、分区表和数据库内置对象。
- 必须允许 NULL、永久保留特殊数据、生产数据测试或非标准字符集的场景。
- 在线 DDL 算法、锁等待、主从延迟、执行窗口及失败恢复方式。

## 来源例外勿复制

- 不把 PDF 中的示例 DDL 当作合规模板；示例缺少五个基础字段且字符集与正文要求不一致。
- “所有整型统一 INT”与“使用最小适配类型”存在冲突；按真实范围选择类型并在偏离既有模型时评审。
- `TIMESTAMP`、默认值、NULL 语义、统计查询和多表查询必须结合 MySQL 版本及业务语义，不机械套用草稿措辞。
- 不为满足索引数量上限而删除仍被关键 SQL 使用的索引，也不为每个条件列机械新增索引。
